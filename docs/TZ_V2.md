# Техническое задание v2: универсальный Agent Context Hub

Статус: проект требований  
Дата: 12 июня 2026 года  
Текущее рабочее название: Waymark Hub  
Будущее название: определяется после стабилизации продукта

## 1. Назначение

Создать независимый от модели и поставщика координационный слой для совместной
работы AI-агентов над проектами.

Hub должен позволять новому агенту быстро понять:

- над каким проектом ведётся работа;
- что уже было сделано;
- какие решения приняты и почему;
- какие задачи активны или ожидают исполнителя;
- где находятся подтверждающие источники;
- с какого места продолжать работу.

Пользователь не должен вручную пересказывать каждому агенту историю проекта.
Агенты должны работать как команда с общей памятью и контролируемой передачей
контекста.

## 2. Продуктовое позиционирование

Agent Context Hub — локальный MCP-сервис общей памяти, координации и передачи
задач между AI-агентами.

Основная ценность:

- уменьшение повторного исследования проекта;
- уменьшение повторных объяснений со стороны пользователя;
- передача работы между разными моделями и клиентами;
- сохранение решений и причин их принятия;
- выдача только релевантного контекста в заданном токен-бюджете;
- измерение фактической пользы и стоимости контекста.

Hub не является самостоятельным агентом и не должен быть привязан к Claude,
Codex, ChatGPT или другому конкретному поставщику.

## 3. Главная проверяемая гипотеза

Использование Hub снижает суммарный расход токенов и время выполнения задач,
которые продолжаются в нескольких сессиях или передаются между агентами, без
снижения качества результата.

Сам факт хранения памяти не считается экономией. В расчёт должны входить:

- токены инструкций и схем MCP-инструментов;
- токены запросов к Hub;
- токены возвращённого контекста;
- токены на создание итогов и записей памяти;
- повторное чтение файлов и документации;
- повторное исследование уже решённых вопросов;
- исправление ошибок, вызванных устаревшим или нерелевантным контекстом.

Основная метрика считается по полному usage всех модельных вызовов в рамках
задачи. Входные токены агента уже включают инструкции, схемы инструментов,
полученный контекст и запросы на запись памяти, поэтому их нельзя прибавлять
повторно.

Формула основной метрики:

```text
net_token_saving =
  total_model_tokens_without_hub
  - (total_agent_model_tokens_with_hub + optional_hub_llm_tokens)
```

`optional_hub_llm_tokens` учитываются только при использовании внешней модели
для сжатия, embeddings или ранжирования. Детерминированные операции локального
сервера токены модели не расходуют.

Токены контекста, схем инструментов и записи памяти дополнительно сохраняются
как диагностические составляющие, но не суммируются с полным usage второй раз.

Относительная экономия:

```text
net_token_saving_percent =
  net_token_saving / tokens_without_hub * 100
```

## 4. Целевые показатели

Показатели должны быть подтверждены A/B-бенчмарком, а не субъективной оценкой.

### 4.1 MVP

- не менее 20% медианной чистой экономии токенов на задачах продолжения;
- не менее 30% сокращения повторно прочитанных файлов;
- не менее 25% сокращения времени до первого полезного действия;
- не более 5% случаев, когда выданный контекст ухудшил результат;
- не менее 90% успешных передач задачи без дополнительного пересказа
  пользователем;
- ответ основных read-only инструментов менее чем за 100 мс на локальной базе
  до 100 000 записей, без учёта времени клиента и модели.

### 4.2 Целевое состояние

- 35–50% медианной экономии токенов на многосессионных задачах;
- не менее 80% выданных записей отмечаются агентами как использованные;
- менее 2% критически устаревших или ошибочных записей в выдаче;
- восстановление рабочего контекста нового агента за один вызов Hub.

## 5. Границы системы

### 5.1 Hub обязан

- хранить проекты, память, решения, задачи, сессии и обратную связь;
- идентифицировать клиента, агента, модель и сессию без привязки к поставщику;
- формировать компактный контекст под конкретную задачу;
- соблюдать заданный бюджет контекста;
- сохранять происхождение и актуальность информации;
- поддерживать stdio и защищённый локальный HTTP-транспорт;
- предоставлять измерения использования Hub;
- обеспечивать миграции данных без потери существующей базы.

### 5.2 Hub не обязан

- заменять Git, issue tracker или проектную документацию;
- хранить полный текст каждого диалога;
- автоматически видеть действия агента, если клиент их не передал;
- гарантировать живую синхронизацию контекста внутри уже запущенной сессии;
- напрямую вызывать конкретную LLM в обязательном порядке;
- хранить секреты, токены API и содержимое `.env`;
- самостоятельно изменять исходный код проектов.

## 6. Архитектурные принципы

1. **Provider-agnostic.** В ядре нет логики, обязательной только для одного
   поставщика или клиента.
2. **Context on demand.** Полные записи не загружаются автоматически без
   необходимости.
3. **Budget first.** Любая агрегированная выдача имеет ограничение по размеру.
4. **Evidence over retelling.** Предпочтительны ссылки на файлы, коммиты,
   задачи и источники вместо копирования больших текстов.
5. **Deterministic core.** Базовый поиск, фильтрация и ранжирование работают без
   LLM и без сети.
6. **Optional intelligence.** Семантический поиск и LLM-сжатие подключаются
   отдельными адаптерами.
7. **Traceability.** Для записи известно, кто, когда и на основании чего её
   создал.
8. **Replace, do not accumulate.** Устаревшая информация заменяется или
   архивируется, а не бесконечно добавляется.
9. **Local first.** Базовый режим работает полностью локально.
10. **Backward-compatible migrations.** Обновления схемы не требуют удаления
    существующей базы.

## 7. Основные сценарии

### 7.1 Продолжение работы другим агентом

1. Агент A регистрирует сессию и выполняет часть задачи.
2. Агент A фиксирует результат, решения, препятствия и точку продолжения.
3. Агент B открывает проект в другом клиенте или на другой модели.
4. Агент B запрашивает контекст для текущей задачи с токен-бюджетом.
5. Hub возвращает краткий пакет контекста и ссылки на подробности.
6. Агент B продолжает работу без ручного пересказа пользователя.

### 7.2 Делегирование подзадачи

1. Агент создаёт задачу и указывает необходимые возможности исполнителя.
2. Другой агент находит подходящую pending-задачу.
3. Задача атомарно переводится в `in_progress`.
4. Исполнитель сохраняет результат и подтверждающие источники.
5. Исходный агент получает итог через Hub.

### 7.3 Возврат к проекту после перерыва

Hub возвращает:

- активную цель;
- последние существенные изменения;
- открытые задачи и блокеры;
- действующие решения;
- устаревшие записи, которые нельзя использовать;
- рекомендуемую точку продолжения.

### 7.4 Поиск прошлого решения

Агент ищет решение по смыслу или ключевым словам и получает:

- краткий ответ;
- причину решения;
- дату и автора;
- статус актуальности;
- ссылки на файлы, коммиты и заменившие решения.

## 8. Модель данных v2

Окончательная SQL-схема определяется на этапе проектирования миграции.

### 8.1 Agents и clients

Необходимо разделить:

- `provider` — OpenAI, Anthropic, Google, local и другие;
- `model` — идентификатор модели;
- `client` — Codex CLI, Claude Desktop, web agent и другие;
- `agent_id` — логический исполнитель;
- `session_id` — конкретная рабочая сессия;
- `capabilities` — код, браузер, документы, исследования и другие возможности.

Поле `surface` должно быть постепенно заменено этой моделью. На переходном
этапе оно сохраняется для обратной совместимости.

### 8.2 Memory records

Минимальные поля:

- `id`;
- `project_id`;
- `kind`: `fact`, `decision`, `summary`, `handoff`, `reference`, `preference`;
- `title`;
- `summary`;
- `body`;
- `status`: `active`, `superseded`, `stale`, `archived`;
- `importance`: 0–100;
- `confidence`: 0–100;
- `source_type`;
- `source_ref`;
- `created_by_agent`;
- `created_in_session`;
- `valid_from`;
- `valid_until`;
- `supersedes_id`;
- `created_at`;
- `updated_at`;
- `last_verified_at`.

Короткое `summary` используется для индекса и компактной выдачи. Полное `body`
загружается только по необходимости.

### 8.3 Context packets

Сформированный пакет должен содержать:

- идентификатор проекта;
- формулировку текущей задачи;
- активную цель;
- краткое состояние;
- релевантные решения и факты;
- активные задачи и блокеры;
- точку продолжения;
- ссылки на подробные записи;
- приблизительный размер в токенах;
- объяснение, почему записи включены в пакет.

### 8.4 Feedback

Для каждой выданной записи агент может отметить:

- `used`;
- `not_used`;
- `helpful`;
- `irrelevant`;
- `stale`;
- `incorrect`;
- `too_verbose`.

Feedback должен использоваться для аналитики и последующего ранжирования.

### 8.5 Usage и benchmarks

Необходимо хранить:

- поставщика, модель, клиента и сессию;
- входные и выходные токены, если клиент их передал;
- оценку токенов контекста Hub;
- количество вызовов инструментов;
- прочитанные файлы, если клиент их передал;
- длительность задачи;
- результат и оценку качества;
- принадлежность к A/B-эксперименту.

## 9. MCP API v2

Названия предварительные и уточняются перед реализацией.

### 9.1 Компактный старт

```text
workspace_resume(
  project_id,
  task?,
  agent?,
  max_tokens=1200
)
```

Один основной вызов для восстановления состояния. Он должен заменить
обязательную последовательность `project_list + memory_list + task_list`.

### 9.2 Контекст под задачу

```text
context_get(
  project_id,
  task,
  max_tokens,
  kinds?,
  include_sources=false
)
```

Выдача должна быть ограничена бюджетом и отсортирована по полезности.

### 9.3 Запись результата

```text
session_finish(
  session_id,
  outcome,
  summary,
  decisions?,
  handoff?,
  files_touched?,
  commits?,
  usage?
)
```

Один вызов должен закрывать сессию и фиксировать структурированный handoff без
необходимости отдельно вызывать несколько инструментов.

### 9.4 Задачи

Нужны операции:

- создание;
- поиск по проекту, статусу и требуемым возможностям;
- атомарный claim;
- обновление прогресса;
- завершение;
- отмена;
- повторное открытие с очисткой `completed_at`;
- фиксация зависимостей и блокеров.

### 9.5 Память

Сохраняются низкоуровневые операции чтения, поиска и записи для совместимости и
отладки. Они не должны быть рекомендуемым способом старта каждой сессии.

### 9.6 Телеметрия

```text
usage_report(
  session_id,
  input_tokens?,
  output_tokens?,
  cached_tokens?,
  duration_ms?,
  files_read?,
  result_quality?
)
```

Поля usage являются необязательными, поскольку не каждый MCP-клиент раскрывает
статистику модели.

## 10. Формирование контекста

### 10.1 Базовое ранжирование без LLM

Приоритет рассчитывается на основе:

- совпадения проекта;
- текстовой релевантности задаче;
- типа записи;
- важности;
- актуальности;
- статуса;
- давности проверки;
- положительной и отрицательной обратной связи;
- связи с активной задачей;
- наличия подтверждающего источника.

Устаревшие и заменённые записи по умолчанию не входят в пакет.

### 10.2 Токен-бюджет

Hub должен:

1. оценить размер обязательной части;
2. зарезервировать место для активной цели, handoff и блокеров;
3. добавлять остальные записи по рейтингу;
4. не превышать бюджет более чем на 10%;
5. отдавать ссылки на исключённые подробности;
6. сообщать фактическую оценку размера пакета.

Токенизация должна поддерживать подключаемые адаптеры моделей. При отсутствии
адаптера используется документированная приблизительная оценка.

### 10.3 Сжатие

В MVP используется сохранённое автором краткое `summary`.

Автоматическое LLM-сжатие допускается после MVP как необязательный адаптер.
Исходный текст нельзя заменять сгенерированным резюме без сохранения оригинала.

## 11. Исследование экономии токенов

### 11.1 Наборы задач

Минимальный набор:

- продолжение исправления бага в новой сессии;
- передача задачи другой модели;
- возвращение к проекту через несколько дней;
- поиск ранее принятого архитектурного решения;
- браузерная проверка после изменений кода;
- параллельная работа двух агентов над независимыми подзадачами.

### 11.2 Эксперимент

Для каждого сценария:

1. зафиксировать исходное состояние проекта;
2. выполнить задачу без Hub;
3. восстановить исходное состояние;
4. выполнить задачу с Hub;
5. использовать одинаковую модель и максимально одинаковые инструкции;
6. повторить не менее пяти раз;
7. сравнить медиану, разброс и качество.

При сравнении разных моделей результаты группируются по модели. Нельзя
приписывать Hub экономию, вызванную различиями моделей.

### 11.3 Метрики

- полные input/output tokens;
- оценка токенов схем MCP;
- токены полученного и записанного контекста;
- количество tool calls;
- количество уникальных и повторно прочитанных файлов;
- время до первого полезного изменения;
- полное время задачи;
- число запросов уточнения пользователю;
- успешность продолжения без пересказа;
- качество результата по заранее заданной проверке;
- количество использованных и неиспользованных записей.

### 11.4 Ограничения измерений

Если клиент не передаёт usage модели, результаты отмечаются как оценочные.
Оценочные и точные измерения нельзя смешивать без отдельной маркировки.

## 12. Безопасность

- HTTP по умолчанию слушает только loopback-интерфейс.
- Для внешнего доступа требуется явная настройка bind address.
- Обязательны Host/Origin validation и документированная аутентификация.
- В память запрещено автоматически записывать секреты и содержимое `.env`.
- Необходимы ограничения размера входных полей и результатов.
- Все SQL-запросы используют параметры.
- Ошибки SQLite и FTS не должны раскрывать внутренние детали клиенту.
- Операции записи и claim задач должны быть транзакционными.
- Должны быть предусмотрены экспорт, резервное копирование и восстановление.

## 13. Совместимость и миграция

- Существующие проекты, memory nodes, tasks и sessions сохраняются.
- Старые MCP-инструменты остаются доступными минимум один стабильный релиз.
- Новые поля добавляются миграциями.
- `surface` преобразуется в совместимую запись client/agent.
- Существующие типы памяти отображаются на новые `kind`.
- До переименования сохраняются текущие package name, пути и connector id.
- Переименование выполняется отдельным этапом после стабилизации API v2.

## 14. Нефункциональные требования

- Node.js 22+ и TypeScript strict mode.
- SQLite остаётся базовым локальным хранилищем.
- Работа без сети и без API-ключей в базовой конфигурации.
- Автоматические unit и integration tests.
- Изолированные временные базы для тестов.
- Проверка stdio и HTTP transport в CI.
- Структурированные диагностические логи без содержимого секретов.
- Предсказуемые ошибки MCP с кодом и кратким сообщением.
- Документированная политика хранения и очистки данных.

## 15. Этапы реализации

### Этап 0. Стабилизация текущей версии

- исправить HTTP bind и Host/Origin protection;
- исправить приоритет проектной памяти над глобальной;
- очищать `completed_at` при повторном открытии задачи;
- добавить обработку ошибок FTS;
- добавить тестовый раннер и integration tests;
- проверить жизненный цикл HTTP transport;
- добавить резервное копирование тестовой и рабочей базы.

Результат: текущая функциональность надёжна и покрыта тестами.

### Этап 1. Измерительный фундамент

- добавить сущности agents, clients, sessions usage и experiments;
- реализовать `usage_report`;
- добавить оценку токенов контекста;
- создать CLI или скрипт запуска A/B-бенчмарков;
- зафиксировать baseline без Hub.

Результат: можно доказательно определить, экономит Hub токены или расходует их.

### Этап 2. Универсальная модель агентов

- убрать Claude/Codex-специфичные ограничения из схемы;
- добавить provider/model/client/agent/session;
- добавить capabilities;
- сохранить обратную совместимость.

Результат: разные модели и клиенты представлены одинаково.

### Этап 3. Компактный контекст

- реализовать `workspace_resume`;
- реализовать `context_get`;
- добавить token budget;
- добавить детерминированное ранжирование;
- возвращать краткие записи и ссылки на подробности.

Результат: новый агент восстанавливает состояние одним компактным вызовом.

### Этап 4. Жизненный цикл памяти

- добавить статусы, актуальность, confidence и provenance;
- реализовать supersede и archive;
- добавить feedback;
- добавить отчёты о неиспользуемой и устаревшей памяти;
- реализовать дедупликацию.

Результат: память не растёт бесконтрольно и остаётся пригодной для работы.

### Этап 5. Командная координация

- атомарный claim задач;
- зависимости, блокеры и требуемые capabilities;
- структурированный handoff;
- защита от одновременного исполнения одной задачи;
- обзор активности агентов.

Результат: Hub поддерживает реальную параллельную работу команды агентов.

### Этап 6. Опциональный интеллектуальный слой

- semantic search;
- адаптеры embeddings;
- адаптеры LLM summarization;
- model-specific tokenizers;
- обучаемое ранжирование на feedback.

Результат: качество выборки повышается без обязательной зависимости ядра от LLM.

### Этап 7. Переименование и выпуск

- выбрать нейтральное название;
- мигрировать package name, server name и connector documentation;
- предоставить aliases и инструкцию обновления;
- подготовить публичную документацию и примеры интеграций.

## 16. Критерии приёмки MVP

MVP включает минимально необходимые результаты этапов 0–5. Этапы 6 и 7 не
входят в MVP: интеллектуальные адаптеры и переименование выполняются только
после подтверждения основной гипотезы.

MVP считается готовым, если:

1. текущие подтверждённые дефекты исправлены и покрыты тестами;
2. новая сессия получает контекст проекта одним вызовом;
3. выдача соблюдает указанный token budget;
4. проектные записи имеют приоритет над глобальными;
5. устаревшие и superseded записи не попадают в обычную выдачу;
6. handoff содержит завершённое, незавершённое, блокеры и точку продолжения;
7. задача может быть безопасно захвачена только одним агентом;
8. usage и оценка токенов сохраняются по сессиям;
9. A/B-бенчмарк воспроизводим;
10. подтверждена медианная чистая экономия не менее 20% на задачах продолжения;
11. миграция существующей базы проходит без потери записей;
12. stdio и локальный HTTP режимы проходят integration tests.

## 17. Риски

### Раздувание контекста

Мера: token budget, компактные summary, выдача подробностей по ссылке.

### Устаревшая память

Мера: статусы, valid_until, last_verified_at, supersede и feedback.

### Низкое качество handoff

Мера: структурированная схема и обязательные поля точки продолжения.

### Различия между моделями

Мера: provider-agnostic API, адаптеры токенизации и раздельные метрики.

### Недоступность точного usage

Мера: поддержка точных и оценочных измерений с явной маркировкой.

### Слишком сложный MCP API

Мера: несколько высокоуровневых инструментов для обычной работы и
низкоуровневые инструменты только для совместимости и диагностики.

### Зависимость от автоматического summarization

Мера: детерминированное ядро; LLM-функции остаются необязательными.

## 18. Открытые продуктовые решения

Перед реализацией соответствующих этапов необходимо отдельно утвердить:

- формат идентификации агента между независимыми клиентами;
- правила автоматического архивирования;
- точную формулу ранжирования;
- допустимый формат источников и ссылок;
- политику хранения usage и session history;
- необходимость web-интерфейса;
- необходимость удалённого многопользовательского режима;
- поставщиков semantic search и summarization;
- новое нейтральное название продукта.

## 19. Ближайший практический шаг

Начать с этапа 0 и измерительного фундамента этапа 1. До получения baseline не
добавлять semantic search, обязательное LLM-сжатие или сложное автоматическое
ранжирование: сначала необходимо измерить текущую стоимость Hub и доказать,
какие изменения действительно уменьшают расход токенов.
